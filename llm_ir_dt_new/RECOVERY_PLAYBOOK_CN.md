# `llm_ir_dt_new` 恢复操作手册

本文档给出当前仓库在不修改现有 Python 脚本前提下的实际恢复流程，用于实验执行、论文描述和结果复现。

## 1. 适用范围

本项目是一个面向自主事件响应研究的 Docker 化数字孪生实验环境，不是真实生产系统。

因此，这里的“恢复”定义为：

- 保留现有可导出的证据
- 停止受影响实验环境
- 重新构建镜像
- 重新部署一套干净的数字孪生环境
- 验证基础服务恢复正常

这里的恢复不等同于：

- 法证级磁盘镜像采集
- 内存取证
- 精细化单主机隔离
- 生产系统备份恢复
- 永久修补漏洞后恢复上线

## 2. 实验环境角色

当前环境包含两段网络：

- 客户端网络 `10.0.1.0/24`
- 服务端网络 `10.0.2.0/24`

主要主机如下：

- `gateway`：网关与 Snort IDS
- `client`：攻击平台
- `server_ssh`：弱口令 SSH 主机
- `server_samba`：Samba 漏洞主机
- `server_shellshock`：Shellshock 漏洞主机
- `server_web1`：正常 Web 主机
- `server_web2`：正常 Web 主机

## 3. 当前可执行的恢复思路

如果实验已经出现攻击行为，当前仓库最实际的恢复路径是：

1. 保留已有告警与主机证据
2. 停止当前数字孪生环境
3. 重新构建所有镜像
4. 重新部署干净容器
5. 对核心服务执行基础健康检查
6. 清理旧告警，准备下一轮实验

这个流程已经由仓库中的 [run_recovery.py](D:/RA%20Project/AI%20Agent&Cybersecurity%20paper/llm_recovery/llm_ir_dt_new/run_recovery.py) 实现。

## 4. 标准操作流程

### 4.1 启动环境

如果环境尚未启动，可以执行：

```powershell
python .\start.py
```

作用：

- 构建 Docker 镜像
- 创建网络
- 启动容器
- 配置静态路由
- 在网关中启动 Snort

### 4.2 执行攻击或验证流量

基础功能验证：

```powershell
python .\run_command.py
```

完整攻击链执行：

```powershell
python .\run_attack.py
```

作用：

- 产生可观测攻击流量
- 触发 SSH、SMB、HTTP、Shellshock 等告警
- 为后续恢复与取证提供实验数据

### 4.3 执行恢复

当实验结束或需要恢复到干净状态时，执行：

```powershell
python .\run_recovery.py
```

该脚本会自动完成以下动作：

- 导出当前状态信息
- 导出 Snort 告警
- 导出若干主机上的日志和命令输出
- 停止并删除当前数字孪生资源
- 重建镜像
- 重新部署新环境
- 执行基础验证
- 清空告警文件

## 5. 恢复阶段导出的证据

恢复脚本会在 `artifacts/recovery_<timestamp>/` 下生成证据目录。

通常包含：

- `status.json`
- `alerts.json`
- `alerts.raw`
- 各容器子目录中的 `.json` 和 `.txt` 结果

### 5.1 可导出的主机

当前恢复脚本重点导出以下容器证据：

- `client`
- `server_ssh`
- `server_samba`
- `server_shellshock`

### 5.2 可导出的内容

包括但不限于：

- `ip addr`
- `ip route`
- `ps aux`
- `/etc/passwd`
- SSH 日志
- Samba 日志
- Apache 访问日志
- Apache 错误日志
- CGI 文件内容
- 文件共享目录列表

### 5.3 不包括的内容

当前仓库不支持：

- 磁盘镜像采集
- 内存镜像采集
- 只读介质写入
- 证据链条管理

因此这里更准确地说是“实验取证导出”，不是严格意义上的“法证采集”。

## 6. 恢复后的验证

`run_recovery.py` 会自动进行基础验证，通常包括：

- 数字孪生整体状态检查
- `client -> server_ssh` 连通性测试
- `client -> server_web1` HTTP 测试
- `client -> server_web2` HTTP 测试
- `server_ssh` 上 `/home/admin` 目录检查

如果这些检查通过，可以认为：

- 网络基础连通正常
- Web 服务已恢复
- SSH 主机已恢复
- 实验环境已回到可用基线

## 7. 如何理解“恢复成功”

在这个项目里，“恢复成功”意味着：

- 原攻击过程中产生的运行态痕迹已被清除
- 容器被替换为干净的新实例
- 网络拓扑重新建立
- 基础服务重新可达
- 可以开始下一轮攻击与检测实验

这不意味着：

- 所有漏洞已修补
- 所有弱口令已修复
- 环境已经适合上线生产

因为这些脆弱点本身就是实验设计的一部分。

## 8. 与通用 IR 流程的对应关系

如果从更标准的事件响应角度看，本项目对应关系如下：

- 检测：由 Snort 告警和实验流量完成
- 遏制：通过停止环境或直接进入恢复流程近似实现
- 取证：通过导出日志和命令输出来近似实现
- 根除：通过删除旧容器并重建镜像实现
- 恢复：通过重新部署和健康检查实现

其中最弱的一环是：

- 精细化遏制
- 深度取证
- 自动化分析

## 11. 当前版本的结论

在不修改现有 `.py` 文件的前提下，本仓库已经具备一条完整、可复现的实验恢复路径：

- 可检测攻击
- 可导出部分证据
- 可销毁受影响环境
- 可重建干净环境
- 可验证恢复结果

因此，对于本项目的研究目标而言，答案是：

- 可以恢复系统
- 但恢复的是“实验基线环境”，不是“生产安全加固后的业务系统”
